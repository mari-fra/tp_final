Este trabajo presenta un análisis comparativo de diferentes modelos de Machine Learning aplicados al problema de clasificación propuesto por Mercado Libre en la competencia de Kaggle (Meli Challenge 2019). El objetivo principal es evaluar y comparar el rendimiento de diversos modelos contra la solución ganadora.

El Meli Challenge 2019 fue una competencia de Kaggle organizada por Mercado Libre con el objetivo de clasificar productos en diferentes categorías basándose en sus títulos y descripciones. La solución ganadora que alcanzó un score de 0.91733 en el Public Leaderboard, ocupando el primer lugar.

\section{Marco Teórico}

\subsection{Machine Learning para Clasificación de Texto}

La clasificación de texto es una tarea fundamental en el procesamiento de lenguaje natural (NLP) que consiste en asignar etiquetas o categorías a documentos de texto. En el contexto del Meli Challenge 2019, el objetivo es clasificar productos de e-commerce en categorías específicas basándose en sus descripciones.

En la literatura es habitual organizar los enfoques de clasificación de texto en tres grandes grupos. Los \emph{enfoques tradicionales} incluyen representaciones como TF-IDF o FastText con clasificadores clásicos (por ejemplo, regresión logística, SVM o árboles). El \emph{aprendizaje profundo} abarca redes recurrentes (LSTM, GRU) y convolucionales aplicadas al texto. Los \emph{transformers} corresponden a arquitecturas basadas en mecanismos de atención (BERT, BETO, GPT). Esta taxonomía es la que se adopta en el presente trabajo para comparar modelos.

\subsection{Arquitecturas de Redes Neuronales}

% Habla sobre LSTM, GRU, transformers, etc.
Las arquitecturas de redes neuronales profundas han demostrado ser efectivas para tareas de clasificación de texto. Entre las más utilizadas se encuentran:

\begin{itemize}
    \item \textbf{LSTM (Long Short-Term Memory):} Redes neuronales recurrentes capaces de capturar dependencias de largo plazo en secuencias de texto
    \item \textbf{GRU (Gated Recurrent Unit):} Variante más eficiente computacionalmente que LSTM
    \item \textbf{Transformers:} Arquitecturas basadas en mecanismos de atención que han revolucionado el NLP moderno
    \item \textbf{Embeddings:} Representaciones vectoriales densas de palabras o tokens que capturan relaciones semánticas
\end{itemize}

\subsection{Embeddings y modelos tipo Transformer}

Los embeddings de texto utilizados en clasificación, búsqueda y bases vectoriales suelen producirse con \emph{encoders} basados en la arquitectura Transformer. Esta subsección resume los conceptos y etapas típicas de ese pipeline, siguiendo la exposición de Gutsch\footnote{Gutsch, D. (2023). \textit{The Art of Embeddings: Transforming Text for Vector Databases (Part 2)}. Medium. \url{https://medium.com/@david.gutsch0/the-art-of-embeddings-transforming-text-for-vector-databases-926738443e70}}. Estos conceptos sustentan, en mayor o menor medida, modelos como BERT y los embeddings de APIs como la de OpenAI utilizados en la etapa de laboratorio (Modelo 3).

\subsubsection{Tokenización}

La tokenización es el proceso de dividir el texto en unidades más pequeñas o \emph{tokens}. Los tokens pueden ser palabras, subpalabras o caracteres. En modelos clásicos como Word2Vec se usaba vocabulario a nivel de palabra, lo que limita el manejo de palabras fuera de vocabulario (OOV) y hace crecer el modelo con el tamaño del vocabulario.

Entre las técnicas de \emph{subpalabras} utilizadas en este proyecto cabe destacar dos familias. Por un lado, \textbf{FastText} (Modelo 4, FastText) define las subpalabras como \emph{n-gramas de caracteres} de cada palabra (p. ej. ``celular'' $\to$ ``cel'', ``elu'', ``lul'', ``ula'', ``lar''). El vector de la palabra se obtiene como la suma de los vectores de sus n-gramas; así se pueden representar palabras OOV que comparten n-gramas con el corpus de entrenamiento. Por otro lado, los modelos tipo Transformer (BERT, APIs como OpenAI) usan \emph{tokenización por subpalabras} con esquemas como Byte Pair Encoding (BPE) o WordPiece: las palabras frecuentes se mantienen enteras y las raras se descomponen en subunidades, con vocabularios del orden de 30k--50k tokens (p. ej. en BERT o GPT-2). En ambos casos se controla el tamaño del vocabulario y se mejora el manejo de OOV. La elección del esquema de tokenización afecta el tamaño del modelo, el manejo de OOV y la calidad final de los embeddings.

\subsubsection{Embedding lookup (búsqueda de embeddings)}

Una vez tokenizado el texto, cada token se mapea a un vector denso inicial mediante una tabla de embeddings (embedding lookup). Estos vectores no son aleatorios: se aprenden en la fase de pre-entrenamiento del modelo y constituyen el punto de partida para que las capas posteriores capturen el significado de cada token en contexto.

\subsubsection{Codificación posicional (positional encoding)}

La arquitectura Transformer no tiene noción intrínseca del orden de los tokens. Para incorporar la posición en la secuencia, se suman \emph{codificaciones posicionales} a los embeddings iniciales. Así el modelo puede distinguir, por ejemplo, ``casa blanca'' de ``blanca casa''. Esta etapa permite procesar la secuencia de forma no secuencial (a diferencia de RNN/LSTM), lo que facilita el paralelismo y el entrenamiento a escala.

\subsubsection{Mecanismo de autoatención (self-attention)}

El mecanismo de autoatención permite que cada token ``consulte'' al resto de los tokens de la secuencia y pondere su influencia según la relevancia. Así se obtienen representaciones que dependen del contexto completo (p. ej. la misma palabra tiene distinto vector según la oración). En la práctica, para cada posición se calcula una combinación ponderada de las representaciones de todas las posiciones; los pesos se obtienen a partir de productos escalares entre vectores de consulta y clave (attention scores), normalizados típicamente con softmax.

\subsubsection{Agregación y pooling}

Dentro de una capa Transformer, la salida del mecanismo de atención se agrega (p. ej. se suma con residual y se normaliza) y se obtiene un vector por token que ya incorpora contexto. Para tareas que requieren \emph{un único vector por documento o por título} (como clasificación o búsqueda por similitud), se aplica una etapa adicional de \emph{pooling} sobre los vectores de los tokens: por ejemplo, promediar los vectores de todos los tokens (mean pooling), o usar el vector asociado a un token especial como \texttt{[CLS]} en BERT. Ese vector único es el embedding que se indexa o se usa como entrada a un clasificador.

\subsubsection{Red neuronal feed-forward}

Tras las operaciones de atención (y opcionalmente más capas apiladas), los vectores por token se pasan por una red neuronal feed-forward (capas lineales con no linealidad), compartida para todas las posiciones. Esta red refina las representaciones y contribuye a la capacidad del modelo de capturar patrones más abstractos. La salida final de la última capa (antes o después del pooling, según el diseño) es el embedding que se utiliza para comparación o clasificación.

En resumen, el flujo típico es: \emph{tokenización} $\to$ \emph{embedding lookup} $\to$ \emph{+ codificación posicional} $\to$ (repetido) \emph{autoatención} + \emph{agregación/residual} + \emph{feed-forward} $\to$ \emph{pooling} (si se requiere un vector por texto). Los modelos de embedding como \texttt{text-embedding-3-small} de OpenAI no divulgan su arquitectura interna, pero se apoyan en principios análogos para producir vectores densos a partir de texto.

\subsection{OpenAI Embeddings (Modelo 3)}

Se utilizó el modelo \texttt{text-embedding-3-small} de OpenAI para el problema de clasificación de categorías de productos. Esta sección describe la metodología del embedding, la solución implementada, los dos ensayos realizados y el uso de la Batch API.

\subsubsection{Metodología del embedding (text-embedding-3-small)}

Según la documentación de OpenAI\footnote{\url{https://developers.openai.com/api/docs/models/text-embedding-3-small}}, los embeddings son representaciones numéricas de texto que permiten medir la afinidad semántica entre dos fragmentos. El modelo \texttt{text-embedding-3-small} es una versión mejorada y más eficiente del modelo anterior \texttt{ada}; está pensado para búsqueda, clustering, recomendaciones, detección de anomalías y tareas de clasificación.

La API de embeddings recibe como entrada el texto (en este caso, el título del producto) y devuelve un único vector denso de dimensión fija (1536 para \texttt{text-embedding-3-small}). Ese vector resume el contenido semántico del texto en una representación de tamaño fijo apta para entrenar clasificadores estándar. Se utiliza la API para obtener el vector de embedding para cada ejemplo mediante llamadas síncronas. La documentación pública no detalla el pipeline interno (p. ej. tokenización, representación por subpalabras o agregación en un único vector).

\subsubsection{Descripción de la solución}

La solución consiste en: (1) obtener un vector de embedding por título mediante la API de OpenAI; (2) usar esos vectores como features para entrenar clasificadores supervisados. Se probaron Random Forest y K-Nearest Neighbors (KNN), con variantes de KNN: sin escalado, con StandardScaler y con reducción de dimensionalidad (PCA a 50 componentes). La evaluación se realiza sobre un conjunto de test estratificado; las métricas reportadas son accuracy (y, donde aplica, classification report).

\subsubsection{Ensayo 1 (03\_tp\_final\_embedding.ipynb)}

\begin{itemize}
    \item \textbf{Muestra:} 1.000 registros tomados al azar del dataset; filtro de al menos 3 ejemplos por categoría, resultando en 247 filas útiles.
    \item \textbf{API:} Llamadas síncronas a la API de embeddings (no Batch).
    \item \textbf{Train/valid:} 70\% entrenamiento, 30\% validación (172 y 75 muestras respectivamente).
    \item \textbf{Mejor resultado:} KNN con escalado (StandardScaler), accuracy 0,53 en validación.
\end{itemize}

\subsubsection{Ensayo 2 (03\_tp\_final\_embedding\_2.ipynb)}

\begin{itemize}
    \item \textbf{Muestra:} 10.000 registros con muestreo estratificado por categoría; filtro técnico de al menos 20 ejemplos por categoría (alineado con 02\_tp\_final\_beto\_all) y luego filtro de al menos 2 ejemplos por clase para permitir split estratificado (70\% train / 30\% test).
    \item \textbf{API:} Batch API de OpenAI: mismo modelo y misma lógica de embedding, con procesamiento asíncrono y descuento del 50\% en coste\footnote{\url{https://developers.openai.com/api/docs/guides/batch}}.
    \item \textbf{Train/test:} 70\% entrenamiento, 30\% test (\textasciitilde6.880 y \textasciitilde2.949 muestras).
    \item \textbf{Mejor resultado:} KNN sin escalado, accuracy 0,48 en test; Random Forest 0,19; KNN con PCA 0,38.
\end{itemize}

La accuracy no mejoró en el Ensayo 2 a pesar de usar una muestra mayor (10k frente a \textasciitilde247) porque el problema es más difícil: en el Ensayo 2 se conservan muchas más categorías (con mínimo 20 ejemplos en el dataset completo y luego 2 en la muestra), de modo que el número de clases es mucho mayor. A mayor número de clases, la tarea de clasificación es más difícil y la accuracy típicamente baja para el mismo modelo y la misma métrica; además, el Ensayo 1 evaluaba en un conjunto de validación pequeño y con pocas categorías, por lo que la comparación directa no es equitativa. La metodología del Ensayo 2 (Batch API, split train/test, filtros por categoría) es más rigurosa para reportar resultados.

\subsubsection{Batch API}

La Batch API de OpenAI\footnote{\url{https://developers.openai.com/api/docs/guides/batch}} permite enviar conjuntos de peticiones de forma asíncrona. Para el endpoint \texttt{/v1/embeddings}, los parámetros de cada petición (modelo, texto de entrada) son los mismos que en la API síncrona; la diferencia está en el modo de envío y el coste. Ventajas relevantes para este proyecto: (1) descuento del 50\% respecto a la API estándar; (2) límites de tasa separados y más altos; (3) tiempo de completitud dentro de 24 horas (suele ser menor). Cada lote se prepara en un archivo JSONL (una línea por petición, con \texttt{custom\_id}, \texttt{method}, \texttt{url} y \texttt{body}); el archivo se sube con \texttt{purpose="batch"} y se crea el batch indicando \texttt{endpoint="/v1/embeddings"}. Los resultados se descargan por \texttt{output\_file\_id}; el orden de las líneas puede no coincidir con el de entrada, por lo que se usa \texttt{custom\_id} para asociar cada respuesta al título correspondiente. Para embeddings, el límite es de 50.000 entradas por lote; en el Ensayo 2 se usaron 10.000 títulos en un solo lote.

\subsection{BETO Aplicado al Dataset Completo en Español}

Para evaluar el rendimiento del modelo BETO en condiciones más representativas, se implementó una versión que utiliza el dataset completo en español (\textasciitilde10 millones de registros) en lugar de las muestras reducidas utilizadas en experimentos preliminares.

\subsubsection{Configuración del Entrenamiento}

El entrenamiento se realizó en un equipo con GPU NVIDIA RTX 3060 (12GB VRAM), lo cual requirió configuraciones específicas para optimizar el uso de memoria y acelerar el proceso:

\begin{itemize}
    \item \textbf{Mixed Precision (FP16):} Reducción del uso de VRAM en aproximadamente 40\% y aceleración del entrenamiento en \textasciitilde20\%
    \item \textbf{Batch size:} 16 por dispositivo con gradient accumulation de 2 pasos (batch efectivo de 32)
    \item \textbf{Warmup ratio:} 10\% de los steps totales para estabilizar el entrenamiento inicial
    \item \textbf{Early stopping:} Paciencia de 2 epochs para evitar overfitting
\end{itemize}

\subsubsection{Dependencia de Accelerate}

Un aspecto técnico relevante es la dependencia de la librería \texttt{accelerate} de Hugging Face. El \texttt{Trainer} de Hugging Face Transformers delega todas las operaciones de bajo nivel a esta librería, incluyendo:

\begin{itemize}
    \item Movimiento de tensores entre CPU y GPU
    \item Implementación de mixed precision (FP16/BF16)
    \item Manejo de gradient accumulation
    \item Soporte para entrenamiento distribuido multi-GPU
\end{itemize}

Esta separación arquitectónica se realizó para mantener la librería \texttt{transformers} más liviana, permitir configuraciones avanzadas sin modificar el código base, y reutilizar la lógica de aceleración en otras librerías del ecosistema Hugging Face (como \texttt{diffusers}).

\subsubsection{Preparación del Dataset}

Para el split estratificado del dataset se requirió un filtro mínimo de 20 ejemplos por categoría. Esto se debe a que el proceso de división (90\% train, 5\% validación, 5\% test) con estratificación necesita suficientes ejemplos en cada categoría para garantizar representación en todos los conjuntos. Categorías con menos de 20 ejemplos pueden resultar en splits donde alguna categoría quede con solo 1 ejemplo, causando errores en la estratificación.

\subsubsection{Requisitos de Software}

La ejecución en GPU moderna (RTX 30xx/40xx) con CUDA 12.x requiere:
\begin{itemize}
    \item PyTorch $\geq$ 2.6 (debido a vulnerabilidad de seguridad CVE-2025-32434 en versiones anteriores)
    \item \texttt{accelerate} $\geq$ 0.26.0
    \item Drivers NVIDIA actualizados compatibles con CUDA 12.x
\end{itemize}

\subsubsection{Validación de Extrapolabilidad de Resultados}

Con el propósito de validar si los resultados obtenidos con una muestra reducida de 70,000 registros (notebook \texttt{02\_tp\_final\_beto.ipynb}) eran extrapolables al dataset completo en español, se entrenó el mismo modelo BETO con el dataset completo de aproximadamente 10 millones de registros (notebook \texttt{02\_tp\_final\_beto\_all.ipynb}) utilizando la misma arquitectura y configuración de hiperparámetros.

Los resultados obtenidos muestran una diferencia en el rendimiento entre ambas configuraciones:

\begin{itemize}
    \item \textbf{Muestra de 70,000 registros:} Accuracy de 0.46 en el conjunto de test
    \item \textbf{Dataset completo (\textasciitilde10M registros):} Accuracy de 0.88 en el conjunto de test
\end{itemize}

La diferencia observada de 42 puntos porcentuales indica que los resultados no son extrapolables de la muestra pequeña al dataset completo. Las posibles razones de esta discrepancia incluyen:

\begin{enumerate}
    \item \textbf{Representatividad de la muestra:} Una muestra de 70,000 registros puede no capturar adecuadamente la distribución completa de categorías y patrones presentes en el dataset completo, considerando que el dataset original contiene más de 1,500 categorías diferentes.
    
    \item \textbf{Balance de clases:} Con una muestra pequeña, algunas categorías pueden estar subrepresentadas o ausentes, afectando la capacidad del modelo para aprender patrones generalizables.
    
    \item \textbf{Overfitting en muestra pequeña:} Con menos datos, el modelo puede memorizar patrones específicos de la muestra en lugar de aprender representaciones generalizables, resultando en un rendimiento reducido cuando se evalúa en el conjunto completo.
    
    \item \textbf{Capacidad del modelo:} BETO es un modelo de 110 millones de parámetros que requiere grandes cantidades de datos para alcanzar su potencial. Con 70,000 ejemplos, el modelo no tiene suficientes datos para aprender efectivamente las relaciones complejas entre texto y categorías.
\end{enumerate}

Esta validación demuestra la importancia de evaluar modelos en datasets de tamaño similar al de producción, especialmente cuando se trabaja con modelos transformer de gran capacidad como BETO. Los resultados preliminares con muestras pequeñas pueden no reflejar el rendimiento del modelo en condiciones de producción.

\section{Problemas Técnicos Encontrados}

\subsection{Limitaciones de la Solución Original}

La solución ganadora del 2019 presenta problemas de reproducibilidad debido a:

\begin{enumerate}
    \item \textbf{Dependencias obsoletas:}
    \begin{itemize}
        \item Python 3.6 / 3.7 (EOL - End of Life)
        \item TensorFlow 1.14
        \item CUDA 10.0 + cuDNN 7.6
    \end{itemize}
    
    \item \textbf{Incompatibilidades técnicas:}
    \begin{itemize}
        \item TensorFlow 1.14 no es compatible con GPUs modernas (RTX 30xx)
        \item CuDNNLSTM y CuDNNGRU requieren versiones específicas de CUDA/cuDNN que ya no están disponibles para drivers modernos
        \item Error: ``No OpKernel was registered to support Op 'CudnnRNN''
        \item Estas capas no tienen fallback a CPU, por lo que el entrenamiento falla inmediatamente sin GPU funcional
    \end{itemize}
    
    \item \textbf{Problemas con bibliotecas:}
    \begin{itemize}
        \item Incompatibilidad gensim + smart\_open
        \item SyntaxError con Python 3.6 por uso de \texttt{from \_\_future\_\_ import annotations}
        \item El paquete smart\_open utiliza anotaciones de tipo de Python 3.10+, no soportadas en Python 3.6
    \end{itemize}
    
    \item \textbf{Recursos computacionales:}
    \begin{itemize}
        \item Embeddings de gran tamaño (\textasciitilde6 GB)
        \item FastText requiere mucha RAM durante el entrenamiento
        \item Entrenamiento extremadamente lento en CPU
        \item Costo computacional elevado incluso con las tecnologías modernas
    \end{itemize}
\end{enumerate}

\subsection{Workarounds Implementados}

Para poder evaluar los modelos, se implementaron las siguientes soluciones alternativas:

\begin{itemize}
    \item Reemplazo de CuDNNLSTM/CuDNNGRU por LSTM/GRU estándar (permitiendo ejecución en CPU pero perdiendo la aceleración GPU específica de la solución original)
    \item Migración a TensorFlow 2.x o alternativas modernas
    \item Adaptación del código para ejecución en CPU
    \item Uso de versiones modernas de Python (3.8+) y bibliotecas compatibles
\end{itemize}

\textbf{Nota importante:} Estos workarounds permiten la ejecución funcional pero alteran la arquitectura original, haciendo imposible una replicación exacta de la solución ganadora del 2019 en entornos modernos.
